# 1. Описание процесса
• Какие города выбрали и почему?

  Для отчета были выбраны города Москва, Санкт-Петербург, Сочи, Казань, Новосибирск. Данные города удалены друг от друга, находятся в разных регионах России и имеют разные климатические условия для сравнения.

• Какие проблемы встретили при сборе данных?

  При сборе сырых данных основными вызовами стали ограничения бесплатного API и риск перезаписи файлов при повторном запуске на следующий день.

• Какие правила очистки применили?

  Для обеспечения надёжности внедрены: безопасный парсинг полей,разделение времени сбора и времени измерения, а также версионирование файлов по дате, реализована структура хранения по дате data/raw/openweather_api/YYYY/MM/DD/.

# 3. Архитектурные решения

• Почему выбрали именно такую структуру папок?
  - Классическая цепочка слоев Raw, Cleaned, Enriched, Aggregated. Разделение по слоям обработки изолирует сырые ответы API от бизнес-логики, позволяет перезапускать отдельные этапы без потери исходников, упрощает отладку и аудит, логирование позволяет отслеживать весь процесс по шагам и быстро локализовать ошибки.

• Какие метаданные считаете самыми важными?
  - collection_time и dt (время скрипта и время измерения сервером) в json файлах важные метаданные
  - логирование позволяет отслеживать весь процесс по шагам и быстро локализовать ошибки.

• Как бы масштабировали решение на 100 городов?
  - возможно вместо csv, сохранения данных в PostgreSQL для экономии памяти и быстрыой агрегации

# 4. Выводы и улучшения

• Что можно улучшить в пайплайне?

  - сохранения данных в PostgreSQL для экономии памяти и быстрыой агрегации
  - асинхронные запросы (aiohttp) для сокращения времени сбора

• Какие дополнительные данные хотели бы добавить?

  Потенциально хотелось еще добавить:
  - события, фестивали
  - индекс загрязнения воздуха
  - курс валют

• Как автоматизировать процесс?

  Можно так же добавить автоматический запуск каждый день, визуализация на графиках для менеджеров, хранение в PostgreSQL истории за все время
